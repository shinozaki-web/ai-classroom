## 0. このプロジェクトの本質

**北九州市の子供たちのAIスキルを、日本一のレベルに育てる。**

これは「全国向けに売れる汎用AI教材」を作るプロジェクトではない。
**北九州市内（門司区・小倉北区・小倉南区・八幡東区・八幡西区・戸畑区・若松区）への展開**を横展開のゴールとする、地域密着・地域限定の本気のカリキュラムである。

汎用商品として広く浅く展開すれば、ベネッセや大手プログラミングスクールと同じ土俵で戦うことになり、勝ち筋がない。
北九州市という単位で本気を尽くせば、競合はほぼゼロであり、地方創生×AI教育という物語そのものが「日本一」級の実績になる。

北九州ゆるキャラ（ていたん・じーも・ヤハタン・すなQ）＋下関せきまるをバッジモチーフに使う設計は、この地域限定戦略と完全に整合している。

## 1. 教育哲学（揺らがない軸）

- **「AIは現代の文房具」** — 魔法ではなく、準備とスキルを要する実践的な道具
- 教えるのではなく**体感させる**。自分でAIを動かした体験を最重視する
- AIを使う子は集中力が約2倍になる（詰まった時に自己解決できる環境が集中を持続させる）という観察知見がある
- AIを使いこなせる子の3つの素質：①「次に何を聞くべきか」を自分で探せる（問いを立てる力）②AI生成物のクセやハルシネーションを前提理解する感覚③「所詮AIだよね」と思いながら遊び相手として使い倒す姿勢
- タイピングは「AI時代のそろばん」。音声入力が普及してもタイピングを教えるのは、そろばんと電卓の関係に近い
- プログラミング基礎＋AI活用＋タイピング技術の3本柱。タイピングが上手な子ほどAIを使う能力（AIにAIを使わせる能力）が上がる
- 教育で重要なのは「過去のこと」ではなく「今リアルに必要なこと」と「これから必要になること」（旧式サイドブレーキの車校教習の例え）
- 人材とスキルの地産地消。地元の子供が地域課題をAIで解決する経験を通じて、地元に住み続ける理由を作る

## 2. 全体ロードマップ（小4〜高2、計5〜6年）

```
小4─────小5─────小6 │ 中1─────中2─────中3 │ 高1─────高2
└─ フェーズ1-3 ─────┘ └─ 同カリキュラム天井up ┘ └─ Claude Code版 ──┘
  (AI教育の基礎)         (チャレンジで深掘り)      (社会課題Webアプリ公開)
```

### 2-1. 小学生フェーズ（4〜6年、フェーズ1〜3、全14回想定）

| ブロック | 回数 | テーマ | 内容 |
|---|---|---|---|
| ①AIと仲良くなる | 第1〜4回 | 体験と対話 | AIとは何か／聞き方の工夫（プロンプト基礎）／役割を与える／AIに情報を教える |
| ②AIと一緒に作る | 第5〜10回 | 制作 | 詩・クイズ制作／地域調査／発表資料／自由制作×2回 |
| ③AIを問い直す | 第11〜14回 | 批評・社会応用 | ハルシネーション／著作権・倫理／地域課題への応用／チャレンジ（中学生はここから深掘り） |

**マイルストーン：**
- フェーズ1終了時：自分でプロンプトを書いてAIと会話できる
- フェーズ2終了時：AIを使ってテーマを決めた作品を完成できる
- フェーズ3終了時（=小学生卒業時）：AIを道具として使いこなし、社会に活かせる

### 2-2. 中学生フェーズ（同カリキュラムの天井を上げる）

小学生と同じ枠組み・同じ教材を使うが、**各回に「チャレンジミッション」を必ず設定**し、★〜★★★の段階で深掘りできるようにする。
小学生から上がってきた子は物足りなさを感じず、中学から始めた子も無理なくキャッチアップできる二層構造。

チャレンジ例の質：「なぜClaudeは悪いことを断るの？その判断を誰が決めてるの？」のような、AI開発者視点に立たせる問い。説教にせず、思考させる設計にする。

### 2-3. 高校生フェーズ（Claude Code版・別カリキュラム）

- **部活動的な立ち位置**で運営（コワーキングスペースを拠点に毎週集まる）
- 数人のチームで**1つの社会課題を解決するWebアプリ**をClaude Codeで実際に実装
- 高1〜高2の2年間で完成・公開（高3は受験のため実質的な活動期間ではない）
- 無料/有料は問わず、**実際に社会に公開する**ことがゴール
- 継続のカギ：「毎週集まれる場所と時間」＋「篠崎さんが伴走できる体制」＋「場所がある・大人がいる・定期的に来る義務がある」の3点セット

## 3. 成果物ロードマップ（段階的ブラッシュアップ設計）

**設計思想：ジャンルは自由、複雑さの階段は共通。**
子供の興味に応じてジャンルを選ばせ、学期ごとに変えてもよい。ただし「どれだけAIと協働する力が育っているか」の複雑さの軸は学年で共通して上げていく。

### 3-1. 成果物ジャンル例（A〜Gの引き出し）

| ジャンル | 例 |
|---|---|
| A. 物語・表現系 | 自己紹介／地域の歴史を物語に→脚本→小説→アニメーション |
| B. 調査・分析系 | 人口データ分析／地域課題の調査／政策提案レポート |
| C. ものづくり・道具系 | お小遣い計算アプリ／時間割管理／地域店舗紹介アプリ |
| D. ゲーム・エンタメ系 | クイズゲーム／自作ルールのミニゲーム／対戦ゲーム |
| E. 観察・記録系 | 給食の好き嫌い記録／植物の成長日記／自由研究 |
| F. デザイン・クリエイティブ系 | 行事ポスター／ゆるキャラグッズ／地域ロゴ提案 |
| G. 社会課題解決系 | 困りごとの聞き取り→プロトタイプ→Webアプリ公開（高校生フェーズに直結） |

### 3-2. 複雑さの階段（全ジャンル共通の軸）

| 段階 | 学年 | AIとの関わり方 |
|---|---|---|
| 種をまく | 小4 | AIに聞く・短いものを一緒に作る |
| 形にする | 小5 | テーマを自分で決め、AIと往復しながら形にする |
| 演出する | 小6 | 複数の要素を組み合わせる（文章＋画像、データ＋グラフ等） |
| 設計する | 中1〜2 | 構成・設計を自分で考え、AIに任せる部分と自分でやる部分を切り分ける |
| 創作する | 中3 | 人に見せる前提で完成度を仕上げる、批評をもらって改善する |
| 公開する | 高1〜2 | 実際に社会で使われる形（Webアプリ等）にして公開する |

### 3-3. 成長の定点観測

先生用ダッシュボードに「過去の成果物アーカイブ」機能を持たせ、子供自身が「去年の自分」と「今の自分」を比較できるようにする。これが「能力向上」の最も説得力ある証明になる。

## 4. 発表会・外部接点の設計

「日本一」を名乗るには、外に見せて証明する場が必須。継続のモチベーションにも直結する。

| タイミング | 発表の場 | 規模 | 狙い |
|---|---|---|---|
| 学期末（年3回） | 教室内発表会 | 生徒同士・保護者 | 小さな成功体験の積み重ね |
| 年度末 | 門司港エリアでの展示・発表イベント | 地域住民・他区の子も招待 | 地域への発信、他区への種まき |
| 中学生以降 | 北九州市内の各区交流発表会 | 各区から集まる | 「各区への横展開」の自然な入口 |
| 高校生 | 公開Webアプリのお披露目会 | 地域メディア・行政・地域企業も招待 | 社会的証明、実績化 |

年度末発表会を「各区への横展開の入口」として設計する。発表会に他区の子を呼ぶ→翌年度から正式に展開、という流れを作る。

## 5. 運営体制についての考え方（現時点では拡張しない）

カリキュラムは現時点では**篠崎さん個人の哲学・判断・伴走に強く依存する設計**でよい。これは欠陥ではなく意図的な選択である。

「篠崎さん以外でも回る」ための指導者育成・マニュアル化・フランチャイズ化は、**今は目指さない**。
このカリキュラムの魅力に本気で共鳴する同志が自然に集まってくることを前提とし、無理に再現性・汎用性を作って篠崎さんらしさを薄めない。

ただし将来的に北九州市全体に展開するフェーズが来た場合、この前提は見直しが必要になることは記録しておく。

## 6. 現在の技術構成（実装済み）

```
ai-classroom/
├── index.html       # トップ画面・授業選択・進捗管理
├── shared.css        # 共通デザイン
├── shared.js          # 共通ロジック（進捗保存・バッジ判定など）
├── lesson1.html       # 第1回：AIってなんだろう？Claudeと話してみよう
├── lesson2.html       # 第2回：聞き方を変えると変わる（プロンプトの工夫）
└── teacher.html       # 先生用ダッシュボード（クラスコード管理・進捗確認）
```

- **公開URL：** https://shinozaki-web.github.io/ai-classroom/
- **リポジトリ：** shinozaki-web/ai-classroom（GitHub Pages有効化済み）
- **データ：** localStorageで完結。ログイン不要、クラスコード方式で進捗を先生が確認できる
- **Claude連携：** アプリ内にAPIキーを持たせず、claude.aiを別タブで開かせる方式（教育的に「本物のツールに触れる」ことを重視）

### 6-1. 設計上の重要な原則

- **「答えを教科書に書かない」**：子供が実際にclaude.aiを開いてプロンプトを送り、結果を持ち帰って比較する体験型にする。受動的な読み物にしない（第2回STEP2で実装済みのパターン）
- **必須ミッション＋チャレンジミッションの2層構造**：チャレンジ側を厚くすることで、小学生でも中学生でも同じ教材で対応できる「天井の高さ」を確保する
- **バッジ制度**：北九州ゆるキャラ（ていたん・じーも・ヤハタン・すなQ）＋下関せきまるで、ステップ完了ごとに収集型の達成感を作る
- **構造化を維持する**：1ファイルに全部詰め込まず、`lessonN.html`を独立させ、共通部分は`shared.css`/`shared.js`に集約する。修正のたびに全体を読み直す無駄を避ける（トークン効率・保守性の両立）

## 7. 当面のアクションプラン

1. 小学生向けWebアプリの各回（lesson3.html以降）を、教室での実施に合わせて順次作成する
2. 各回を作るたびに「教室専用の調整」か「汎用化できる構造」かを意識して分ける
3. 中学生向けチャレンジミッションのレベル感を、実際の中学生の反応を見ながら調整する
4. 高校生向けClaude Code版カリキュラムは別途設計する（このドキュメントの範囲外、着手は小中が一定形になった後）
5. 発表会・成果物アーカイブ機能は、教室運営が安定してから先生用ダッシュボードに組み込む
